# Canonical Event Envelope and Vocabulary

Status: canonical Step 6a event foundation

Parent issues:

- `#35` Step 6a: Ship the Core Evidence Loop
- `#36` Step 6a.1: Define the Canonical Event Envelope and Vocabulary

## Purpose

This document defines the event contract that Step 6a replay, persistence, scoring, and comparison work must share.

It is intentionally narrower than a full observability platform. The goal is to make run evidence:

- normalized across native and hosted execution
- durable enough for replay and comparison
- explicit about evidence quality and degradation
- versioned from day one

## Core Rule

`run_events` stores canonical event rows, not provider-specific or adapter-specific event names.

Every event written for Step 6a must answer:

- what happened
- when it happened
- which `Run` and `RunAgent` it belongs to
- which emitter/source produced it
- whether the evidence is rich structured telemetry or a degraded black-box observation

## Canonical Envelope

The code-level definition lives in `backend/internal/runevents`.

Required envelope fields:

- `event_id`: idempotency-oriented event identity. For hosted callbacks this should be deterministic from callback data. For native execution this may be generated by the emitter.
- `schema_version`: current value `2026-03-15`
- `run_id`
- `run_agent_id`
- `sequence_number`: append-only per `run_agent_id`; assigned at durable insert time
- `event_type`: canonical vocabulary entry
- `source`: emitter identity such as `native_engine` or `hosted_external`
- `occurred_at`
- `payload`: event-specific JSON payload
- `summary`: compact query/index metadata for replay builders and future compare/scoring reads

Important sequencing rule:

- `sequence_number` is authoritative for replay order inside one `RunAgent`
- `occurred_at` is descriptive time, not the conflict resolver
- pre-persistence normalization may produce an envelope with `sequence_number = 0`; it becomes valid for durable storage only after the repository assigns a positive sequence

## Source Vocabulary

Current canonical sources:

- `native_engine`: native step loop and sandbox-driven execution
- `hosted_external`: hosted agent callback data normalized into AgentClash vocabulary
- `hosted_callback`: reserved for API/control-plane callback lifecycle events if they need separate persistence later
- `worker_scoring`: scorecard and metric emission

The current database column is still named `actor_type`; Step 6a should treat it as the persisted source field.

## Event Type Vocabulary

Canonical event types are grouped by responsibility, not by provider.

### System events

- `system.run.started`
- `system.run.completed`
- `system.run.failed`
- `system.output.finalized`
- `system.step.started`
- `system.step.completed`

### Model events

- `model.call.started`
- `model.call.completed`
- `model.output.delta`
- `model.tool_calls.proposed`

### Tool events

- `tool.call.started`
- `tool.call.completed`
- `tool.call.failed`

### Sandbox events

- `sandbox.command.started`
- `sandbox.command.completed`
- `sandbox.command.failed`
- `sandbox.file.read`
- `sandbox.file.written`
- `sandbox.file.listed`

### Scoring events

- `scoring.started`
- `scoring.metric.recorded`
- `scoring.completed`
- `scoring.failed`

## Summary Metadata

`summary` is not a second payload blob. It is the small set of indexing fields likely to matter for replay and comparison.

Current summary metadata fields:

- `status`
- `step_index`
- `provider_key`
- `provider_model_id`
- `tool_name`
- `sandbox_action`
- `metric_key`
- `external_run_id`
- `evidence_level`
- `idempotency_key`

Guidance:

- keep summary compact and stable
- store filterable/index-worthy metadata here
- keep large raw fragments, provider dumps, and diagnostic blobs in `payload` or future artifact storage

## Native vs Hosted Normalization

Native and hosted execution do not have the same fidelity. The canonical contract must preserve that fact.

### Native execution

Native execution is expected to emit rich structured events:

- step boundaries
- provider call start/completion
- tool call lifecycle
- sandbox file/command operations

Native events should usually carry `evidence_level = native_structured`.

### Hosted execution

Hosted execution may be either:

- `hosted_black_box`: only coarse lifecycle/final answer/error signals are available
- `hosted_structured`: hosted metadata contains structured trace fragments, but still not full native-fidelity sandbox/tool evidence

Hosted degradation rule:

- do not fabricate model/tool/sandbox events when the hosted system did not provide that evidence
- map coarse hosted events into the nearest canonical system event instead

Current hosted callback mapping:

- `run_started` -> `system.run.started`
- `final_answer` -> `system.output.finalized`
- `error` -> `system.run.failed`
- `run_finished` with `completed` -> `system.run.completed`
- `run_finished` with `failed` -> `system.run.failed`

## Idempotency Rules

Step 6a needs deterministic ingestion behavior even before a full dedupe index exists.

Rules:

- every normalized event must have an `event_id`
- hosted callbacks should derive `event_id` deterministically from callback identity data such as `run_agent_id`, `external_run_id`, raw hosted event type, `occurred_at`, and final status when present
- `summary.idempotency_key` should mirror the same stable identity
- later repository work may add explicit unique indexes or side tables, but the identity contract starts here

Important limitation:

- the current `run_events` table enforces unique `(run_agent_id, sequence_number)`, not global event id uniqueness
- follow-on persistence work should add stronger duplicate protection if replay ingestion starts seeing retries or redeliveries in practice

## Storage Boundaries

Store relationally in `run_events` columns:

- `run_id`
- `run_agent_id`
- `sequence_number`
- `event_type`
- `source` via the current `actor_type` column
- `occurred_at`

Store inside `payload`:

- raw provider or hosted fragments
- final output snippets
- tool arguments/results
- error details
- diagnostic metadata
- raw upstream event type when normalization occurred

Move to artifact/object storage later when payloads become large:

- full raw provider responses
- full streamed token traces
- large tool outputs
- sandbox stdout/stderr blobs
- replay asset bundles

## Package Placement

Step 6a foundation code should live in `backend/internal/runevents`.

Why this package exists:

- `provider` should not own replay vocabulary
- `engine` should not own hosted normalization
- `repository` should persist canonical events, not define them
- replay/scoring work in later issues needs one shared import path for event types and envelope rules

## Immediate Follow-On Guidance

Issue `#37` and later work should build on this contract in this order:

1. persist canonical native events using the shared vocabulary
2. keep hosted callback ingestion writing the same vocabulary
3. build replay indexes from canonical events, not execution-specific event names
4. let scorecards and comparison read evidence quality explicitly rather than inferring it from source package details
