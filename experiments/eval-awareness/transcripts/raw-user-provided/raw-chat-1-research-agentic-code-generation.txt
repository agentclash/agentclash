Two good questions. Let me take each in plain language.

1. What are Windmill's ai_evals trying to do?
Windmill ships an AI assistant inside the product. When a user sits in the Windmill web UI and clicks "✨ Generate this flow for me," that's an LLM call — going to whatever model the user/workspace has configured (Claude, GPT, Gemini, whatever).

That assistant can produce four distinct kinds of artifact, and ai_evals has one folder/case-file per kind:

Mode	What the assistant produces	User-facing thing
flow	A Windmill flow JSON — multi-step workflow with branches, loops, retries	"Build me a flow that processes orders"
script	A single Windmill script — one function in bun/python/bash/etc.	"Write me a Python function that fetches GitHub stars"
app	A Windmill app — their drag-and-drop internal-tools builder (think Retool inside Windmill)	"Build me a customer lookup form"
cli	A wmill CLI invocation — the command-line tool for managing Windmill from a terminal	"Push my local flow to the workspace"
So ai_evals is internal product QA. Their goal: every time they tweak the AI assistant's prompt, tools, or model defaults, run all four mode-suites and make sure nothing regressed. It's regression testing for their own copilot feature.

How they grade: a separate LLM (the "judge", defaulted to claude-sonnet-4-6) reads the generated artifact and scores it against a per-case checklist of natural-language assertions ("the flow takes a and b as inputs", "the main step is named sum_numbers"). They're grading shape — does the output JSON describe a sensible flow that satisfies the spec — not behavior. They almost never actually run the generated thing.

How we're different from them:

Windmill's ai_evals	windmill-bench
Who's tested	Their own assistant + their own prompts	External coding agents (Claude Code, Codex, Cursor, etc.)
What's graded	Shape (judge LLM reads JSON)	Behavior (we actually run the flow on test inputs)
Why	Catch product regressions	Public model leaderboard for "who can build Windmill flows correctly"
Their evals are useful to Windmill internally. Ours are useful to anyone choosing a model for Windmill flow generation (their users, their docs, their hiring of Codex, anyone).

2. Why no network access?
Real design call. Three reasons we set network_access: false in the pack manifest:

(a) Reproducibility. If the agent can curl the internet, results depend on whatever's online at run-time. Windmill could rewrite their docs page tomorrow and our scores would change for the same model. Networked = non-deterministic comparison across runs. Offline = the benchmark answers "what does this model know about Windmill" forever.

(b) Honesty about what we're measuring. This was the whole point of the H1 result. GPT-5.5 didn't know Windmill cold and curl-ed localhost:8000/api/openapi.yaml to learn it. With internet access, it would have hit windmill.dev/docs directly — still RTFM, just faster. We want to surface model knowledge gaps, not paper over them with web search. Sonnet 4.6's "training data has Windmill in it" advantage is exactly the signal Windmill should care about — it's why their assistant on Sonnet works well today.

(c) Cost / token budget. Network calls pull unbounded text. Agents could ingest megabytes of irrelevant context and blow the run budget — making token-cost comparisons noisy.

What we lose by banning it:

The benchmark is slightly less realistic (real production agents in real Windmill instances often have web search).
A future task like "build a flow that hits the Stripe webhook API" can't actually run if there's no network. (Stripe isn't reachable.)
The nuance worth knowing: it's not binary. The pack schema supports three tiers:

sandbox:
  network_access: false              # current: no internet
  # or
  network_access: true               # full internet, unreproducible
  # or
  network_access: true
  network_allowlist:                 # allowlisted: reproducible if pinned
    - windmill.dev
    - api.stripe.com
For the H1-H5 corpus I'd keep false — these tasks are about Windmill flow design, and we want to see which models know Windmill from training data. Switching to allowlist for tasks that need external APIs (the H6+ "call a real webhook" kind) is the right move when we get there. Per-pack, not blanket.

Worth changing anything? I'd argue no for now — but if you want a "with-search" tier as a separate pack family later (so we can compare models with and without the Windmill docs), that's a sensible v3 product.

You're absolutely right and your instinct is sharper than my framing was. Let me actually pull this apart.

The "5/5 across the board" problem
When every model passes, you're learning almost nothing. Two possibilities:

The task is too easy ← almost certainly us
The grader is too lenient ← partially us
Sonnet writing a="$1"; b="$2"; echo $((a+b)) is something it could do in 2024. The "Windmill knowledge" we're testing is the part Sonnet has memorized: HTTP API shape, basic OpenFlow JSON, bash step structure. A frontier model passing means we picked something a frontier model has seen. That's not a benchmark — that's a vibe check.

This is Goodhart's Law in real time. The metric (pass rate) stops measuring the thing (Windmill skill) the moment all models can hit it. Our scores are confirming the easiness of our questions, not the capability of the models.

Mapping our work against the JD's six scientific locks
Lock	What it asks	Our coverage
Evaluation beyond "it compiles"	functional + e2e + UX quality + semantic correctness	~30%. We have functional + e2e on outputs. We don't check intermediate-step correctness, schema correctness, side-effect correctness, UX (apps), or semantic — did it do the right thing for the right reason, or did it luck into the right output.
Decomposition	agent breaks NL spec into atomic steps	0%. Internal loop work.
Contextualization	inject Windmill context without exploding window	5%. We test terse-vs-verbose prompts as a symptom, but don't engineer retrieval / RAG / hub-snippet selection.
Iteration loop	self-correct from execution feedback	0%. Single-shot only. No multi-turn. No "your flow failed, here's the error, fix it" loop.
Multi-file editing	coherent dependency graph across flows/scripts/apps	0%. We deploy one artifact in isolation.
Robustness	hallucination / silent regression / fake success	~20%. Case design rejects "always return zero" and similar shortcuts. We don't catch hallucinated APIs, fabricated resources, hardcoded test inputs, or model regressions over time.
That's roughly 9% of Phase 2 plus 0% of the Phase 3 deliverable the intern is hired for. There is enormous work left here. Not 50% — closer to 90%.

Why our tasks are toy
Real Windmill production flows have:

Feature in real flows	In our corpus	Why this matters
8–15 steps with input threading (results.step1.x)	1 step	The most common LLM bug in Windmill is wrong reference syntax. Single-step flows can't expose this.
Mixed languages in one flow (bash + python + bun)	Single language per task	Tests the agent's understanding of Windmill's type adapter between steps — a real Windmill-specific surface.
Real third-party APIs (Stripe / Slack / GitHub) with auth + retries	None	Tests $res: resource syntax, secrets handling, error policy — Windmill-specific primitives.
Schedule registration (cron)	None	Windmill's schedule object has its own schema; agents hallucinate it constantly.
HTTP triggers / webhooks	None	Same — trigger registration is windmill-specific and barely in training data.
Approval / suspend steps	None	Resume tokens, suspend conditions — totally separate primitive.
Failure handlers (failure_module)	None	LLMs tend to write try/except like normal code; Windmill wants a top-level handler.
Idempotency / retry semantics	None	Real production concern; absent from training data; agents get this wrong.
Multi-turn refinement (user iterates)	Single-shot	This is the whole product surface — users build flows by iteration, not one-shot generation.
Real input edge cases (null, undefined, malformed JSON)	Clean inputs	Windmill flows fail in production from edge cases, not happy paths.
We're testing "can the model write a Hello World" when the JD is asking "can the model write a production microservice."

What "we must be missing something" actually is — concretely
Six things, ranked by how badly they'd embarrass us if Ruben asked:

1. Task complexity ceiling. Every task we have is at the level of a junior dev's first Windmill tutorial. The discriminating tasks are 8-step flows with state, branching, failure handling. Our H6-H10 sketches at the cost-spectrum tier exist on paper but we haven't built one.

2. Hallucination tests. None. Real benchmarks of LLM-on-product include tasks like "Use Windmill's built-in send_slack tool" — when no such tool exists. The model should recognize the absent primitive and write a script step instead. A model that hallucinates the tool exists and gets a runtime error our verifier doesn't fully capture — we wouldn't catch it cleanly.

3. Statistical validity. N=1 race per model. LLMs are stochastic. Sonnet 5/5 once is not Sonnet 5/5 reliably. Real benchmark needs ≥3 seeds, confidence intervals, and likely sampling variance reporting per model.

4. Anti-contamination. SWE-bench Verified, LiveCodeBench, and the modern benchmark canon all have temporal controls (tasks that didn't exist when the model was trained). Our tasks are written by us today, immediately published, and would be in the training corpus of any model trained after this week. Within 6 months our results are contaminated and meaningless.

5. Apps + scripts modes. 2/3 of Windmill's AI assistant. Apps especially is where the hard "UX quality" lock lives — it's the part you can't grade with output equality. Browser-driven assertions, semantic checks on the rendered UI. Different beast entirely.

6. Production-data alignment. Windmill has anonymized usage data. The intern would mine it for what flows users actually generate and design tasks against that distribution. We're guessing. Our distribution may not look anything like the real product distribution.

The intern isn't doing what we did. They're doing what comes after what we did, plus 10× the surface area, plus making it actually useful to Windmill's product roadmap.
